# QM 경영검토 스킬

## Management Review 정의

> 출처: IT Governance Framework - ISO 9001 Surveillance Audit Checklist (https://www.itgov-docs.com/blogs/iso-9001/iso-9001-surveillance-audit-checklist)
> *원문 인용: "A management review is a critical component of the surveillance audit checklist. This involves meeting with key members of management to review the organization's quality management system."*
> *한국어 의역: 경영검토는 감사 체크리스트의 핵심 구성요소로, 경영진 핵심 인원과의 회의를 통해 QMS를 검토하는 활동이다.*

## ISO 9001:2015 9.3절 표준 입력 항목

| 입력 항목 (ISO 표준) | 사용자 환경 매핑 | 데이터 소스 |
|---------|--------|----------|
| 직전 검토의 액션 진행 상황 | 이전 보고의 의사결정 추적 | 과거 보고서 |
| 외부·내부 이슈의 변화 | 라이브 환경 변화, 도구 변경 | 회고록 |
| 품질 목표 달성 정도 | KPI 달성률 | qm-kpi-tracker |
| 제품·서비스 적합성 | TC PASS율, 회귀 발생률 | qm-kpi-tracker |
| 부적합 및 시정 조치 | CAPA 진행 상황 | qm-capa-tracker |
| 모니터링·측정 결과 | 효율성 지표 | qm-kpi-tracker |
| 감사 결과 | Internal Audit 결과 | qm-process-auditor |
| 자원의 적절성 | 2인 체제 한계, 도구 ROI | 종합 |
| 개선 기회 | 개선 권고 | 종합 |

## 청자별 보고서 차별화

### A. 경영진 (C-Level) — 1페이지

**관심 영역**: 비즈니스 임팩트, 라이브 안정성, 비용/ROI, 리스크

**작성 원칙**:
- 1페이지에 수렴
- 핵심 수치 + 신호등 표시 (🟢/🟡/🔴)
- 의사결정·지원 요청 사항 명확
- 기술적 디테일 최소화

**표준 형식**:
```
# QM 월간 품질 리포트 (초안) — YYYY-MM

## 한 줄 요약
이번 달 라이브 품질은 [안정/주의/위험] 상태이며, [핵심 성과]와 [핵심 위험]이 관찰되었습니다.

## 1. 품질 현황 (Health Check)
| 영역 | 상태 | 직전 대비 |
|------|------|---------|
| 라이브 안정성 | 🟢 | ↑ |
| TC 자동화 ROI | 🟡 | ↓ |
| 프로세스 준수 | 🟢 | − |

## 2. 주요 성과 (3개 이내)

## 3. 주요 위험 (3개 이내)

## 4. 의사결정·지원 요청 사항
- [ ] [구체적 요청 1] — 결정자
- [ ] [구체적 요청 2] — 결정자

## 5. 다음 달 계획

> LLM 생성 초안 — QA 팀장 검증 후 공유
```

### B. 외부 부서 (개발/기획/PM) — 1-2페이지

**관심 영역**: 자기 부서와의 인터페이스, 협업 효율, 개선 협업

**작성 원칙**:
- 해당 부서 관련 영역만 발췌
- 비난이 아닌 협업 관점
- 구체적 개선 제안

**부서별 강조 포인트**:

| 부서 | 가장 먼저 보여줄 것 |
|-----|----------------|
| 개발팀 (클라/서버) | 빌드 합격률, 평균 수정 시간, 회귀 발생 |
| 기획팀 | 기획서 ↔ TC 매핑 적합도, 기획 변경 처리 시간 |
| PM/PO | 일감 상태 흐름 적합도, 일정 준수율 |
| 아트 | (해당 영역 발췌) |

### C. QA 파트 내부 — 상세

QM 모든 결과를 그대로 종합한 상세 버전. 다음을 모두 포함:
- KPI 전체 (qm-kpi-tracker)
- CAPA 진행 (qm-capa-tracker)
- 감사 결과 (qm-process-auditor)
- 문서 거버넌스 상태 (qm-doc-governor)

## 의사결정 단위 강조

### 좋은 예 (의사결정 단위로 묶음)

```
## 의사결정·지원 요청

- [ ] 다음 분기 자동화 도구 라이선스 갱신 ($500/월) — CTO 결정 필요
- [ ] 외주 QA 인력 추가 (1명, 1개월) — HR 본부장 승인 필요
- [ ] 신규 KPI 정의서 도입 — QA 본부장 + PM 합의 필요
```

### 나쁜 예 (정보 전달만)

```
## 다양한 사항

- 자동화 ROI가 낮아지고 있습니다.
- 인력이 부족합니다.
- 새 KPI가 필요할 수도 있습니다.
```

## 시각적 표현 규칙

### 신호등 표시

- 🟢 안정 / 정상 / 목표 달성
- 🟡 주의 / 추세 악화 / 일부 미달
- 🔴 위험 / 즉시 조치 필요 / 큰 부적합

### 추세 화살표

- ↑ 개선 (KPI 향상)
- ↓ 악화 (KPI 하락)
- − 유지

### 수치 강조

- **굵은 글씨**: 핵심 수치
- 표: 비교 데이터 (직전 vs 현재)
- 차트: 경영진 보고서에서는 가능한 한 단순하게

## 톤 (Tone)

### 권장
- 사실적
- 데이터 기반
- 협업적
- 의사결정 지향

### 지양
- 비난적 ("OO이 잘못함")
- 추측적 ("아마도", "어쩌면")
- 과장적 ("심각한 위기")
- 책임 회피적 ("우리는 다 했는데...")

## 한계 명시

모든 보고서 말미에 다음을 명시:

```
## 한계 (Limitations)

- 본 보고서는 LLM이 생성한 초안입니다. 인간 검증 필요.
- [데이터 한계]: 부족한 데이터 영역
- [측정 정확도]: 신뢰도가 낮은 KPI 영역
- [기간 한계]: 비교 기간 부족 영역
```

## 보고 빈도 권고 (할루시네이션 방지 명시)

> 다음은 운영 권고안입니다. 조직 합의로 조정 가능.
